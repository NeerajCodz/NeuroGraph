## Key Design Decisions

### Backend Structure

1. **Separation of Concerns**: Clear separation between API, MCP, agents, memory, and integrations
2. **Agent Isolation**: Each agent is a standalone module with no interdependencies
3. **Layer Pattern**: Memory layers have dedicated modules for isolation guarantees
4. **Database Abstraction**: Separate drivers for each database with connection pooling
5. **Event-Driven**: All async operations go through event bus

### Frontend Structure

1. **Component-Based**: Reusable components organized by feature
2. **State Management**: Zustand stores for different concerns
3. **Service Layer**: API calls isolated from components
4. **Type Safety**: Comprehensive TypeScript types
5. **Visualization**: D3.js graph visualization as isolated component

### Docker Organization

1. **Service Separation**: Each service (backend, frontend, Neo4j, PostgreSQL) in separate container
2. **Environment Overrides**: Dev and prod compose files for environment-specific configuration
3. **Volume Management**: Persistent volumes for databases
4. **Network Isolation**: Services communicate through Docker networks

### Naming Conventions

- **Files**: lowercase, hyphen-separated (e.g., `memory-manager.py`)
- **Classes**: PascalCase (e.g., `MemoryManager`)
- **Functions**: snake_case (e.g., `get_user_memory`)
- **Constants**: UPPER_SNAKE_CASE (e.g., `MAX_CONFIDENCE`)
- **Components**: PascalCase (e.g., `ChatContainer.tsx`)
- **Types**: PascalCase (e.g., `ChatMessage`)

## Development Workflow

1. **Initial Setup**: `./scripts/setup.sh` - Installs dependencies, creates .env files
2. **Start Development**: `./scripts/dev.sh` - Starts all services with hot reload
3. **Run Tests**: `cd backend && pytest` or `cd frontend && npm test`
4. **Build Production**: `./scripts/build.sh` - Builds Docker images
5. **Deploy**: `./scripts/deploy.sh` - Deploys to production environment

## Testing Strategy

### Backend

- **Unit Tests**: Test individual functions and classes
- **Integration Tests**: Test API endpoints and database operations
- **E2E Tests**: Test complete user workflows

### Frontend

- **Component Tests**: Test React components in isolation
- **Integration Tests**: Test component interactions
- **E2E Tests**: Playwright tests for full user journeys
